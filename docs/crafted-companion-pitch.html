The Story

How we got here, and why the pivot from v1 to v2 is a strategic advantage rather than a setback.

Act 1 — The Problem

Crafted ERP customers live in a world of operational questions that generic NetSuite can't answer directly:

What NetSuite AI can handle

  • "Show me invoices over $50K"
  • "Top 10 customers by revenue this quarter"
  • "Aged receivables by customer"

What Crafted customers actually ask

  • "What is barrel 092425-02 worth, and trace the cost back to the purchase order"
  • "Are we ready to file this month's TTB? What's blocking us?"
  • "For lot WHS-2024-0847, where did we make or lose money from raw materials through retail?"
  • "Which barrels are 4+ years old and ready to harvest?"

Act 2 — Crafted Intelligence v1 (2024–2025)

We built the right thing at the right time

Crafted Intelligence v1 launched as a standalone AI orchestration layer — custom prompt management, custom tool scripts, custom UI, all Doozy-owned. It proved the concept: craft-beverage customers wanted AI that understood their domain, and they engaged with it.

It included: prompt-engine (playbook management), barrel-intelligence, lot-profitability, inventory-supply, compliance-audit, MRP, and batch-genealogy packages. Each a separate SDF project, each independently deployed.

Act 3 — NetSuite Changes the Game

In late 2025, Oracle released the NetSuite AI Companion — a native SuiteApp providing:

  • A governed prompt library with roles, industries, and categories
  • Atlas prompt records that integrate directly into NetSuite's permissions model
  • A clean library UI with Copy / Send-to-Claude / Send-to-ChatGPT actions
  • Usage analytics (copy counts per prompt, per AI provider)
  • Native integration with the MCP (Model Context Protocol) connector

This changed the strategic calculus. Every Crafted ERP customer would soon have an Oracle-native AI experience by default. The question became: do we compete with it or extend it?

Act 4 — v2: Built on Oracle, Learned the Limits

The v2 attempt

Our first instinct when Oracle shipped Atlas was to extend it — store orchestration metadata in a Doozy extension record FK-linked to Oracle's bundle-locked prompt record. Ride Oracle's rails for governance, roles, and library UX. We shipped this architecture and seeded 12 barrel prompts into the TSTDRV1912378 demo.

Why v2 hit a wall

  • Categories were capped at 7 — Oracle's Atlas category list is bundle-locked. "Barrel Operations" and "Lot Profitability" had to ride under "Manufacturing" with subcategory hacks.
  • No room to add fields — user-facing complexity, collection grouping, execution analytics, and changelog fields couldn't live on the Atlas record; they were trapped in the extension layer and required a JOIN on every read.
  • Fragile at every Oracle release — every update to Atlas threatens our FK query. We'd be permanent downstream-of-Oracle, reacting to schema changes.
  • Identity dilution — "we extend Oracle's library" is a weaker market position than "Crafted Intelligence is its own product that happens to coexist with Oracle's."

Act 5 — v3: Independence (2026-04-17)

The strategic pivot

With zero customers yet deployed on v2, we took the clean moment and rebuilt. Prompt text, categories, roles, and orchestration metadata all live in one Doozy-owned record. The Atlas SuiteApp becomes optional — we surface its prompts read-only in a secondary library tab when it's installed, but we don't depend on it. Crafted Intelligence is its own product.

NetSuite AI Companion 101

Leadership context: what Oracle shipped, why it matters, and what it does and doesn't do.

What It Is

NetSuite AI Companion is Oracle's native AI interface layer for NetSuite. It's a bundle-locked SuiteApp that installs into any NetSuite account and provides a curated library of pre-built prompts that users can invoke through Claude, ChatGPT, or other MCP-compatible AI clients.

The key idea

Instead of every user typing ad-hoc AI queries, the Companion provides a governed library of high-quality prompts — curated by role, industry, and category — that users browse and copy into their AI tool of choice. Oracle calls this "prompt engineering as a first-class enterprise artifact."

What Oracle Ships

Prompt Library~120

Pre-built prompts across ~40 industries, categorized by role. Examples: "Top slow-paying customers", "Aged receivables by segment".

MCP ConnectorTools

A native MCP server that exposes NetSuite data (records, saved searches, SuiteQL, reports) as tools AI assistants can call.

Role GovernanceNative

Prompts are filtered by the current user's role. Admins curate what each role sees.

Usage AnalyticsTracked

Every Copy / Send-to-Claude / Send-to-ChatGPT action is logged per user per prompt.

What It Does Well

  • Generic business questions that ride on standard NetSuite records
  • Standard reports, saved searches, SuiteQL queries
  • Cross-industry knowledge (A/R, A/P, inventory basics, GL)
  • Role-based gating and governance

What It Doesn't Do

  • Industry-specific operational logic — it doesn't know what a proof gallon is, or how to compute angel's share, or what makes a batch "harvest-ready"
  • Custom record orchestration — complex multi-step workflows across barrel records, lot numbers, cellar orders
  • Domain-aware safety rules — e.g., TTB filing readiness requires specific validation not shipped in the base
  • Compliance-grade execution guarantees — beyond generic access control
The Gap We Fill
Oracle ships the platform and the generic prompts. Customers in specialized industries (craft beverage, cannabis, CPG, etc.) need someone to provide the domain intelligence. That someone is the ISV — which in craft beverage is Doozy.

The Orchestration Model

Every Crafted prompt begins with a header that triggers the orchestration handshake:

[Crafted Prompt #201 — call getPromptMeta(201) first for orchestration context, safety rules, and tool chain]

What is barrel RUM1000095 worth? Show me current value, cost per proof gallon,
fill date, age, status, and location. Then trace how the value was built.

When an AI assistant (Claude, ChatGPT) sees this header, it performs a deterministic 5-step flow:

  1. getPromptMeta(201) → returns orchestration envelope: steps, params, safety rules, governance tier, artifact flag
  2. Validate params against user input (substitute [BARREL NUMBER])
  3. Execute the tool chain in order, respecting depends_on
  4. Apply safety rules (e.g., "never fabricate values", "flag zero-cost entries")
  5. logExecution with results, duration, success flag, agent identity

Why This Pattern Matters

Deterministic execution

The AI doesn't improvise the sequence — it follows an orchestration plan Doozy authored. Reduces hallucination risk and makes behavior auditable.

Governance-aware

Four rigor tiers let us match AI autonomy to data sensitivity. A portfolio read is "Minimal"; a compliance readiness check is "Governed".

Example End-to-End

User pastes the "Single Barrel Valuation" prompt into Claude. What happens:

User:  [Crafted Prompt #201 — ...] What is barrel RUM1000095 worth?

Claude: [calls getPromptMeta(201)]
        → reads: governance=Standard, steps=[getBarrelProfile, getBarrelValuation, getBarrelCostTrace]
        → reads: params={BARREL NUMBER: required}, artifact=cost-waterfall

Claude: [extracts "RUM1000095" as BARREL NUMBER]
        [calls getBarrelProfile(barrelSearch: "RUM1000095")]
        → returns: 11 years old, 68.9 PG, 130 proof, at Heaven Hill

Claude: [calls getBarrelValuation(serialNumber: "RUM1000095")]
        → returns: $241.18 current value, $3.50/PG

Claude: [calls getBarrelCostTrace(barrelSearch: "RUM1000095")]
        → returns: Item Receipt 3/10/2023, PO #208, $241.18

Claude: [applies safety rules, generates cost-waterfall artifact]
        [calls logExecution(promptId: 201, success: true, ...)]

Installation Flow (v3)

  1. Doozy deploys Crafted Intelligence SDF bundle via SuiteCloud CLI — creates the independent record, custom lists, 2 custom roles (customrole_dz_ci_admin / customrole_dz_ci_user), 2 Suitelets (Library + Admin Deployment Dashboard), and the Map/Reduce seed script.
  2. Domain toolsets deploy based on customer edition (barrel-intelligence for distilleries, batch-genealogy for wineries, etc.). Prompts requiring missing toolsets are hidden by the Library's hard tool-validation check.
  3. Admin opens the Deployment Dashboard Suitelet — sees a diff (create / update / skip / orphan) between the packaged seed-data.json and current deployed state. Click Preview Role Resolution to verify which account roles the pattern matching will grant access to (e.g. "Administrator", "Controller", "Warehouse"). Click Deploy Prompts to trigger the Map/Reduce.
  4. Map/Reduce runs — creates 37 customrecord_dz_companion_prompt records idempotently (skips up-to-date, updates older versions, creates missing). Role-pattern matching populates visible_roles against the account's real NetSuite roles.

  6. Customer installs the Crafted Intelligence skill file into Claude / ChatGPT / Cowork. If Oracle's AI Companion SuiteApp is also installed, the Library's secondary Oracle tab lights up automatically — no extra configuration.

TTB Compliance — Architectural Complexity

This warrants its own architectural discussion because it's more complex than "query CERP records." Multiple compliance models exist and a given account could be on any of them.

Compliance Has Two Orthogonal Dimensions

Compliance tool behavior is determined by the combination of compliance model and Crafted ERP edition on a given customer account. Both must be probed at runtime; neither can be assumed.

Dimension 1 — Compliance Model (account-level)

The custom list customlist_dz_ic_compliance enumerates how a given Crafted ERP account handles compliance:

Compliance ModelMeaningTool Behavior
NoneAccount has Crafted ERP but Compliance Edition is NOT enabledCompliance tools must detect this and return "not configured" — not fail silently or return zero
LegacyAccount uses the older compliance workflow (custom fields, older saved searches, legacy report records)Tools read legacy workflow definitions + saved search criteria via dz_ct_search_def to drive reports
Total Compliance (list value: "Current")Account uses the modern Crafted ERP Compliance EditionTools use CE_ native records + modern report generators — and still need dz_ct_search_def to read the configured search criteria that drive the reports

The TTB Filing Readiness playbook's Block 1 is a hard gate: if complianceModel = "none" or activeTTBLicenseCount = 0, the analysis stops and escalates to the user with remediation instructions.

Dimension 2 — Crafted ERP Edition

Each edition files a different set of TTB forms. Compliance tools must know the edition (or both, for dual-edition accounts) to determine which filing types apply:

EditionTTB Report TypesNotes
DistilleryMRPDO (Produced Distilled Spirits), MRPCO (Produced Cordials), MRSO (Storage)Proof gallon tracking, DSP license, angel's share analysis
WineryTTB F 5120.17 (Wine Report), excise tax returns, inventory reportsGallons (not PG), batch genealogy, vintage/appellation fields
BreweryTTB F 5130.9 (Brewer's Report of Operations), excise tax returnsBarrels/BBL units, no barrel aging concept

Accounts may be dual-edition (e.g., distillery + winery under one license). Compliance tools must handle both simultaneously.

The Real Matrix — Edition × Compliance Model

Compliance tool behavior is the cross product of these two dimensions. This is the branching logic compliance tools must implement correctly:

Legacy ModeTotal Compliance Mode
DistilleryLegacy distillery workflow + saved searches for MRPDO/MRPCO/MRSOCE_MRPDO/MRPCO/MRSO records + modern report generator + search definitions
WineryLegacy winery workflow + saved searches for wine reportsModern winery compliance records + report generator + search definitions
BreweryLegacy brewery workflow + saved searches for BROModern brewery compliance records + report generator + search definitions

Key insight: dz_ct_search_def is required in both compliance modes, not just Legacy. Even Total Compliance customers configure their reports via saved searches — the difference is what records feed those searches and what report generator consumes them.

Coverage assessment: Option D covers ~70-80% of what dz_intel_config currently caches. The remaining 20-30% (brand profile, full workflow cache) might be unavoidable Doozy-owned infrastructure — but could ship as small targeted records rather than the current intel_config singleton.

Recommended Direction

Pursue C-2 + D as a hybrid:

  • C-2 provides the centralized getAccountConfig tool callable from any prompt
  • D determines what that tool actually probes — start with native detection wherever possible, fall back to small targeted records only for dimensions native detection can't cover

This minimizes Doozy infrastructure while keeping config queries fast and centrally maintained. Decision is captured as a Compliance Deep-Dive Workstream deliverable — needs verification that D's "Probable" coverage rows actually work in practice.

Saved Search Dependency (Required for BOTH Modes)

Both Legacy and Total Compliance customers drive their TTB reports from saved searches with specific filter and formula configurations. The reports are only as trustworthy as the searches feeding them. Compliance tools must:

  • Discover which saved searches exist for compliance filing (by name pattern, script ID, or brand profile reference)
  • Read their search criteria via dz_ct_search_def — understand what filters, formulas, and joins the customer uses
  • Validate that search criteria match what the TTB filing workflow expects (correct date filtering, account filtering, proof/ABV computation, edition-specific field references)
  • Surface discrepancies as data quality issues — before they become filing errors

This is what makes Crafted Companion's compliance layer actually valuable. "Query CUSTOMRECORD_CE_LCN" is the floor; understanding the customer's configured reporting chain is the ceiling.

Risks & Open Questions

What could go wrong, and what we still need to decide.

Strategic Risks

Oracle Framework ChangesLOW

Oracle's AI Companion SuiteApp is bundle-locked. In v2 this was our highest-severity risk — the v3 independence rewrite resolved it at the data layer.

Mitigation (v3): We have no FK dependency on Atlas. The only surface that reads Atlas is the Library's optional secondary tab, which uses try/catch probing and gracefully hides if Atlas schema changes or the SuiteApp is uninstalled. Crafted Intelligence's core data path is untouchable by Oracle updates.

Oracle Enters Craft DirectlyMEDIUM

Oracle could ship industry-specific prompts themselves. Unlikely given craft's specialized vocabulary, but possible.

Mitigation: Our moat is 41 deep domain tools + operational knowledge. Prompts are the surface; tools are the depth. Generic prompts without domain tools aren't competitive.

AI Vendor ShiftsMEDIUM

Customers need Claude/ChatGPT subscriptions. If AI pricing changes drastically, or a vendor consolidates the space, our channel to customers shifts.

Mitigation: MCP protocol is vendor-agnostic. Any MCP-compatible client works. Cowork is another viable channel.

Low AdoptionMEDIUM

Customers buy but don't use. Net revenue retention degrades; renewals drop.

Mitigation: Usage analytics on exec_log to identify at-risk accounts. Quarterly customer success review. Prompt-of-the-month feature for ongoing discovery.

C. Companion Toolset (4 Core Tools, v2.0.0)

  • getPromptMeta(promptId | externalId) — Single-table SELECT against customrecord_dz_companion_prompt. Returns the full v2.0.0 response shape (prompt_text, steps, params, safety_rules, governance, edition, category, description, complexity, collection, related, exec_count, last_executed, avg_duration, changelog). Call first before executing any Crafted prompt.
  • seedPrompt(...) — Idempotent create of a Crafted Intelligence prompt record. Admin use. Supports description, complexity, collection, related, rolePatterns, changelog, and all orchestration fields. Does NOT touch Atlas.
  • updatePrompt(...) — Single record update with auto patch-version bump. Writes directly to the independent record.
  • logExecution(promptId, success, toolsCalled, duration, agent) — Creates exec log row AND writes usage analytics back to the prompt record (increments exec_count, sets last_executed, updates avg_duration via incremental-average formula). Analytics update is best-effort; log success is authoritative. All code paths return Promise.resolve(JSON.stringify({...})).
